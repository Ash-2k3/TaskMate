## Summary

This phase is a pure visual overhaul — no data model changes, no IPC changes, no behavior changes. Every technical question comes down to CSS rendering confidence in Electron's Chromium build, three utility class definitions, a classNames API remap for react-day-picker v9, and plain notification string edits.

The key verified finding is that `backdrop-filter: blur(16px)` works correctly in Electron 41 (Chromium 146) for opaque windows with semi-transparent element backgrounds. The glass surfaces use `rgba(255,255,255,0.05)` backgrounds — this is the necessary transparency for the blur to be visible. No flags, no workarounds needed for this Chromium version. The only Electron-specific caveat is nested `backdrop-filter`: Chromium does not allow child elements to apply their own backdrop-filter if a parent already has one. The plan must order elements so that glass surfaces do not nest inside other glass surfaces in a parent-child DOM relationship.

For the Tailwind build pipeline: Vite 5.4.x and modern autoprefixer already handle the historical `--tw-backdrop-blur` empty-variable stripping bug (fixed in Vite 2.7.13+). Raw CSS `backdrop-filter` in `@layer components` is safer and more explicit than Tailwind utilities for the `.glass` class — it avoids any JIT purge ambiguity entirely and is what the user's discretion area points toward.

**Primary recommendation:** Define `.glass`, `.glass-hover`, and `.transition-glass` as raw CSS in `@layer components`. Use raw CSS `backdrop-filter: blur(16px)` (not `@apply backdrop-blur-2xl`). Override the shadcn Button `default` variant gradient via CSS attribute selector in `@layer components`. Keep all glass surfaces as non-nested siblings, not parent-child, to sidestep the Chromium nested-backdrop-filter limitation.

## Architecture Patterns

### CSS Utility Definition Pattern (`@layer components`)

Define all glass utilities in `index.css` under `@layer components`. Raw CSS, not `@apply`, for backdrop-filter:

```css
/* Source: Tailwind docs @layer components + CSS spec */
@layer components {
  .glass {
    background: rgba(255, 255, 255, 0.05);
    backdrop-filter: blur(16px);
    -webkit-backdrop-filter: blur(16px);
    border: 1px solid rgba(255, 255, 255, 0.10);
  }

  .glass-hover {
    transition: all 0.3s cubic-bezier(0.4, 0, 0.2, 1);
  }

  .glass-hover:hover {
    background: rgba(255, 255, 255, 0.09);
    border-color: rgba(6, 182, 212, 0.25);
    transform: translateY(-2px);
  }

  .transition-glass {
    transition: all 0.3s cubic-bezier(0.4, 0, 0.2, 1);
  }
}
```

### Button Gradient Override Pattern

The shadcn Button `variant="default"` applies `bg-primary` via CVA. Override via CSS attribute/class selector — higher specificity than Tailwind utilities or use `!important` in `@layer components`:

```css
/* Source: shadcn theming docs + CSS cascade */
@layer components {
  /* Override default variant gradient */
  .btn-gradient,
  [data-variant="default"].inline-flex {
    background: linear-gradient(135deg, #06b6d4, #6366f1) !important;
  }
}
```

**Better approach (no !important):** Target by adding a CSS class to the Button className prop site-wide, OR directly edit `button.tsx` to change the `default` variant's Tailwind classes to the gradient token. Since shadcn components are source-owned (not npm-imported), editing `button.tsx` directly is the correct pattern:

```typescript

default: 'bg-primary text-primary-foreground hover:bg-primary/90',
// To:
default: 'bg-gradient-to-br from-cyan-400 to-indigo-500 text-white hover:opacity-90',
```

Or use a raw CSS override in `@layer components` targeting the button's base class combination. The direct `button.tsx` edit is cleanest for this project.

## Common Pitfalls

### Pitfall 1: Nested backdrop-filter breaks in Chromium

**What goes wrong:** If a parent element has `backdrop-filter: blur()` and a child element also has `backdrop-filter: blur()`, Chromium silently ignores the child's filter. Neither renders an error; the child simply shows no blur.

**Why it happens:** Chromium's interpretation of the CSS specification prevents nested backdrop-filter stacking contexts. Firefox and Safari do not have this limitation.

**How to avoid:** Keep glass surfaces as DOM siblings, not parent-child. Specifically: the nav bar (`.glass`) and the page content wrapper must be siblings, not nested. The TodayView header card (`.glass`) and TaskRow elements (`.glass`) must be siblings inside a non-glass wrapper `<div>`, not nested glass inside glass.

**Warning signs:** A glass element renders no blur even though its CSS looks correct; removing a parent class fixes it.

### Pitfall 2: `--border` CSS variable breaks all element borders

**What goes wrong:** The `@layer base { * { @apply border-border; } }` rule in `index.css` applies to every element. If `--border` is set to white (`0 0% 100%`), all elements (inputs, buttons, separators) get a white border.

**Why it happens:** The `border-border` utility resolves to `border-color: hsl(var(--border))` with no opacity modifier. D-06 specifies `--border: 0 0% 100% / 0.10` (white with 10% opacity) but CSS variables do not support alpha in the HSL shorthand the same way Tailwind utilities do.

**How to avoid:** Do NOT set `--border` to `0 0% 100%`. Instead:
1. Keep `--border` at a dark subtle value for non-glass elements (e.g., `240 20% 15%`)
2. Use `rgba(255,255,255,0.10)` directly in the `.glass` class (not via CSS variable)
3. Use `border-white/10` Tailwind utility directly on glass elements where needed

### Pitfall 3: `backdrop-filter` not visible without semi-transparent background

**What goes wrong:** Setting `backdrop-filter: blur(16px)` on an element with a fully opaque background (no alpha) renders nothing — the blur applies to content "behind" the element, which is invisible through an opaque surface.

**Why it happens:** Backdrop-filter only creates a visible effect when the element's own background is transparent or semi-transparent.

**How to avoid:** Always pair `backdrop-filter` with `background: rgba(...)` with alpha < 1. The `.glass` class definition correctly uses `rgba(255,255,255,0.05)`. Never use `bg-card` (solid) on a glass surface.

### Pitfall 4: shadcn Button gradient override specificity war

**What goes wrong:** Adding a CSS rule in `@layer components` to override the Button's `bg-primary` class fails silently because Tailwind utility classes (`bg-primary`) and component layer classes resolve at the same cascade level, but class order in the stylesheet wins — and Tailwind utilities are emitted after `@layer components`.

**Why it happens:** In Tailwind 3.x, `@layer utilities` is processed after `@layer components`, so Tailwind utility classes win over component layer overrides in same-specificity conflicts.

**How to avoid:** Edit `button.tsx` directly (shadcn components are source-owned). Change the `default` variant's CVA string from `bg-primary text-primary-foreground hover:bg-primary/90` to use inline gradient classes. This is cleaner than fighting the CSS cascade. Alternatively, add a wrapper class on the outer `<div>` and target `.btn-primary-gradient button` — higher specificity wins.

### Pitfall 5: react-day-picker v9 `day_hidden` renamed inconsistently

**What goes wrong:** Some migration guides list the v9 replacement as `hidden`; others reference `invisible`. Using the wrong key means outside-month days are either fully hidden (correct) or visible (bug).

**Why it happens:** The v9 API uses `hidden` as a `DayFlag` enum value, but the CSS class string it maps to may differ from the key name.

**How to avoid:** In the remap, use `hidden: 'invisible'` as the classNames entry — the v9 UI enum does not include a `hidden` key, but `DayFlag.Hidden` maps to the classname `'hidden'`. Verify by inspecting the rendered DOM after implementation. If days outside the visible month are not hidden, toggle between `'invisible'` and `'hidden'` as the class value.

### Pitfall 6: Orb `position: fixed` vs `position: absolute` in Electron

**What goes wrong:** If orbs are `position: absolute`, they only extend as far as their parent's height. On short-content pages (empty state), the bottom orb may not appear because the page height is less than the viewport.

**Why it happens:** `position: absolute` is relative to the nearest positioned ancestor; `position: fixed` is relative to the viewport.

**How to avoid:** Always use `position: fixed` for orbs (as specified in D-18/D-19). Confirmed in the App.tsx structure plan.

## Open Questions

1. **`day_hidden` vs `hidden` key in v9 DayFlag**
   - What we know: The v8 key `day_hidden` maps to v9. The `DayFlag` enum includes `Hidden`.
   - What's unclear: Whether the classNames key is `hidden` (matching enum value string) or something else.
   - Recommendation: Use `hidden: 'invisible'` in classNames (matching v8 behavior of hiding the cell with the `invisible` Tailwind class). Verify in rendered DOM after implementation — if outside-month days show, swap to `hidden: 'hidden'`.

2. **`--border` token scope for non-glass elements**
   - What we know: D-06 specifies white/10 for glass borders. The `* { @apply border-border; }` rule applies to all elements.
   - What's unclear: Which non-glass elements will look wrong if `--border` changes to near-white.
   - Recommendation: Keep `--border` at a dark value (`240 20% 15%` or similar), use inline `rgba(255,255,255,0.10)` in `.glass`, and use `border-white/10` Tailwind utility directly on glass elements. Do not route glass border color through the CSS variable.

3. **`nav_button` split in v9**
   - What we know: v8 had single `nav_button` classNames key applying styles to both prev/next buttons.
   - What's unclear: Whether applying the same class to both `button_previous` and `button_next` in v9 is idiomatic, or whether they should be styled differently.
   - Recommendation: Apply identical styles to both `button_previous` and `button_next` since both are ghost outline icon buttons.
