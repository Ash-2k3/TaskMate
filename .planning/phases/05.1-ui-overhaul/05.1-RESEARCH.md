# Phase 05.1: UI Overhaul — Research

**Researched:** 2026-03-23
**Domain:** Desktop app UI design — typography, color, spacing, components, Electron/Tailwind patterns
**Confidence:** HIGH (codebase read directly; design principles from multiple verified sources)

---

## Summary

TaskMate already has a solid structural foundation from Phase 04.1: flat row separators, near-monochrome color, weight-only priority signaling, flush-left forms. The gains from that phase were real. However, reading the actual screen code reveals several areas where the app still feels like a developer-built prototype rather than a polished product.

The highest-leverage gaps are: (1) no Inter font — the app currently uses whatever the browser default picks, which on Electron/macOS is often a mismatched system font; (2) the typography scale collapses too many roles into `text-sm` — headers, body, labels, metadata all look the same weight-and-size; (3) section structure is invisible — WeeklySummary has section headers but no visual breathing room between them; (4) the nav bar is text-only with no visual grounding; (5) the empty state is centered in a flush-left app, breaking the layout language.

The best reference apps (Linear, Craft, Things 3, Notion, Raycast) share one philosophy: **structure should be felt, not seen**. Borders soften. Color is rationed. The font does the hierarchy work. Spacing is generous at section boundaries, tight within sections.

**Primary recommendation:** Add Inter (with `cv11` + `tnum` feature settings), establish a 3-level type scale, increase section-level spacing from `mb-4` to `mb-6`/`mb-8`, ground the nav bar with a slightly more distinct background, and fix the EmptyState layout direction.

## Executive Summary: The 5 Highest-Impact Changes

1. **Add Inter font with OpenType features** — The single biggest visual upgrade. Replaces browser-default font with a typeface designed for UI. The `cv11` (single-story 'a') and `tnum` (tabular numbers for stats) features add professional polish invisible to users but felt in the overall quality.

2. **Establish a 3-level type scale** — Currently nearly everything uses `text-sm`. Introduce `text-[13px]` for metadata/labels, `text-sm` (14px) for body/row content, and a deliberate `text-xs uppercase tracking-wide` pattern for section headers (WeeklySummary already uses this correctly — standardize it everywhere).

3. **Fix section spacing** — WeeklySummary sections use `mb-6` but are crammed together. Add `pb-4 border-b border-border` between major sections to create felt structure without visible noise.

4. **Fix EmptyState layout** — Currently `flex flex-col items-center justify-center h-64`. This puts content in the center of the screen while everything else is flush-left at `px-6`. EmptyState should be flush-left, matching the TaskRow visual baseline.

5. **Tighten the nav bar** — The nav has `py-3` tabs which is fine, but lacks visual grounding. A very subtle `bg-background/95` with a slightly stronger `border-t` makes it feel anchored rather than floating.

## Research Area 1: Typography System

### What premium apps do

**Linear** uses Inter for UI text. Their redesign document confirms: Inter for body, Inter Display for prominent headings. They limit weight usage and use size + weight together to create hierarchy (not just weight alone).

**Tailwind Catalyst** (Tailwind Labs' official premium UI kit) uses:
```css
--font-sans: Inter, sans-serif;
--font-sans--font-feature-settings: 'cv11';
```
`cv11` enables the single-story 'a' which is cleaner for UI text. This is the Tailwind Labs recommendation for Inter in production UI.

**Notion** uses medium font weight (500) as the baseline body, not regular (400). This makes body text feel slightly more confident without being heavy. Warm grays replace harsh blacks.

**Things 3** uses the system font on macOS (SF Pro), but for an Electron app targeting both macOS and Windows, Inter is the correct equivalent — it was designed specifically for screen UI at small sizes.

### Recommended type scale for TaskMate

| Role | Size | Weight | Tailwind | Line Height |
|------|------|--------|----------|-------------|
| Screen title ("Today", "Summary") | 24px | 600 | `text-2xl font-semibold` | `leading-tight` |
| Section subheading ("Week of March 3") | 18px | 500 | `text-lg font-medium` | `leading-snug` |
| Section label (uppercase caps) | 11px | 600 | `text-[11px] font-semibold uppercase tracking-wider` | — |
| Task title / body text | 14px | 400/600 | `text-sm font-normal` / `font-semibold` | `leading-5` |
| Meta / label / date | 13px | 400 | `text-[13px] font-normal` | `leading-4` |
| Overdue badge / helper text | 12px | 400 | `text-xs` | — |

**Note:** The 11px section label pattern (`text-xs uppercase tracking-wider text-muted-foreground`) is already present in WeeklySummary (`text-sm font-semibold text-muted-foreground uppercase tracking-wide`). Reduce the size to 11px and standardize across all screens.

**Current state vs recommended:**
- TodayView header: `text-2xl font-semibold` — CORRECT, keep
- TodayView date subline: `text-sm text-muted-foreground` — should be `text-[13px]`
- TaskRow title: `text-sm` — CORRECT (14px body)
- WeeklySummary section headers: `text-sm font-semibold uppercase tracking-wide` — reduce to `text-[11px]`
- Settings stats: `text-sm` — should be `text-[13px]` for secondary data
- ReflectionsHistory date rows: `font-medium` (inherited) — should explicitly be `text-sm font-medium`

### Inter font implementation

**Install approach (no CDN — offline-first requirement):**
```bash
npm install @fontsource/inter
```

**In `src/renderer/index.css`:**
```css
@import '@fontsource/inter/400.css';
@import '@fontsource/inter/500.css';
@import '@fontsource/inter/600.css';

@layer base {
  :root {
    font-family: 'Inter', system-ui, -apple-system, sans-serif;
    font-feature-settings: 'cv11' 1, 'tnum' 1, 'liga' 1, 'calt' 1;
  }
}
```

**Why these features:**
- `cv11` — single-story 'a', cleaner UI glyphs (Tailwind Catalyst uses this)
- `tnum` — tabular numbers, makes the stats in WeeklySummary align vertically
- `liga` + `calt` — ligatures and contextual alternates, enabled by default on Inter (required for Chrome/Chromium to activate them)

**Tailwind config addition:**
```js
theme: {
  extend: {
    fontFamily: {
      sans: ['Inter', 'system-ui', '-apple-system', 'sans-serif'],
    },
  },
}
```

**Confidence:** HIGH — Inter is the verified standard for Tailwind + shadcn apps. `@fontsource/inter` is the standard offline-safe package. `cv11` + `tnum` pattern comes from official Tailwind Catalyst source.

**Electron font rendering note:** Electron on macOS historically had thin font rendering after macOS 10.14 disabled subpixel antialiasing. Inter is designed with this in mind and renders well at standard weights. Do NOT use weight 300 or below — it renders poorly in Electron's Chromium. Stick to 400, 500, 600. (Source: multiple Electron GitHub issues, verified MEDIUM confidence.)

## Research Area 2: Color & Depth

### What premium apps do

**Linear's philosophy (from their official design refresh post):** "Structure should be felt, not seen." They softened borders rather than removing them. They shifted the default gray from cool/blue-ish to "a warmer gray that still feels crisp, but less saturated." They reduced chrome (accent color) usage to a minimum.

**Notion:** Warm grays instead of pure black/white. Slight differences in text color guide attention without screaming.

**Current TaskMate token analysis:**

```css
--foreground: 222.2 84% 4.9%;    /* Near-black with blue tint — slightly cold */
--muted-foreground: 215.4 16.3% 46.9%;  /* Mid gray */
--border: 214.3 20% 93%;          /* Already lightened in Phase 04.1 — correct */
--muted: 210 40% 96.1%;           /* Very light blue-gray for hover fills */
```

The foreground is slightly cold (blue-tinted near-black). Linear moved toward warmer neutrals. However, changing the base foreground color is low risk and high reward.

### Recommendations

**Warmer foreground (optional but recommended):**
```css
--foreground: 220 15% 8%;   /* Warm near-black, less blue-tinted */
```
This is a subtle shift. If the team wants to stay conservative, leave it. Confidence on exact value: MEDIUM (judgment call).

**Dark mode border:** Current dark mode `--border: 217.2 32.6% 17.5%` is the same as `--muted` — borders are invisible in dark mode. Fix:
```css
.dark {
  --border: 217.2 32.6% 22%;   /* Slightly lighter than muted */
}
```

**Color discipline (already correct from Phase 04.1):**
- Indigo `--primary` on: Add Task button, active nav tab indicator
- Everything else: neutral tokens only
- DO NOT add any new `text-primary` usage

**Confidence:** HIGH for discipline; MEDIUM for exact warmth values.

## Research Area 3: Spacing & Density

### What premium apps do

**Linear** uses a compact row density (~32px total row height) for task lists — achievable with `py-3` (which TaskMate already has — correct). They increase spacing significantly at section boundaries and between major content blocks.

**Notion** uses generous section padding. Between a section header and its content: 8-12px. Between sections: 24-32px.

**Things 3** has tight rows but generous top-of-screen header area with clear visual air between the date/title and the task list.

### Current TaskMate spacing analysis

| Location | Current | Assessment |
|----------|---------|------------|
| Header (`px-6 py-6`) | 24px horizontal, 24px vertical | Good |
| TaskRow (`px-4 py-3`) | 16px horizontal, 12px vertical | Correct — matches Linear density |
| WeeklySummary between sections (`mb-6`) | 24px | Good |
| WeeklySummary section content spacing (`space-y-1`) | 4px between stat rows | Too tight — should be `space-y-2` |
| Settings stats (`space-y-2 mb-8`) | 8px gaps, 32px bottom | Good |
| ReflectionsHistory header (`mb-4`) | 16px | Should be `mb-6` to match other screens |
| TodayView task list wrapper (`mx-6`) | 24px horizontal | Correct |
| AddTask form (`space-y-4`) | 16px between fields | Good |

**Key fix:** WeeklySummary stat rows use `space-y-1` (4px) between "Created / Completed / Rate" lines. At `text-sm`, these lines blur together. `space-y-2` (8px) gives each line visual independence.

**Section separation pattern to standardize:**
```
Section header (text-[11px] uppercase tracking-wider muted)
  ↓ 12px (mb-3)
Section content
  ↓ 24px (mb-6)
Next section header
```

This pattern is already mostly correct in WeeklySummary but not consistent across screens.

**Confidence:** HIGH — derived from direct measurement of existing code against known patterns.

## Research Area 4: Component Language

### Checkbox / completion button

**Current:** `w-4 h-4 rounded-full border border-border` — a circle outline with no fill or checkmark on hover. This works but feels like a wireframe.

**What premium apps do:** Things 3 uses a circular checkbox with a subtle fill on hover. Linear uses a square checkbox with rounded corners. The key differentiator: **the checkbox communicates its affordance on hover** — typically a light fill or slight opacity change on the border.

**Recommended fix (within no-animation constraint):**
```html
<button class="w-4 h-4 rounded-full border border-border flex-shrink-0 hover:border-primary hover:bg-primary/10">
```
This gives the checkbox a clear hover state showing it's interactive, using the one allowed indigo touchpoint in a secondary role (hover only, not persistent).

**Confidence:** MEDIUM — this extends the indigo rule to a secondary hover context. It could be argued this violates the two-touchpoint rule from Phase 04.1. Flag for discussion.

### Back navigation button

**Current:** `text-sm text-muted-foreground hover:text-foreground mb-4` — plain text with no affordance.

**What premium apps do (Craft, Things 3):** Back navigation has a `←` arrow that is visually distinct from body text. The arrow glyph `←` is fine, but it should be sized slightly differently from the label to feel intentional.

**Recommended:**
```html
<button class="flex items-center gap-1 text-[13px] text-muted-foreground hover:text-foreground mb-4">
  <span class="text-base leading-none">←</span>
  <span>Today</span>
</button>
```
This makes the arrow visually larger relative to the label text, matching the hierarchy pattern of back buttons in macOS apps.

### Section header labels (WeeklySummary pattern)

**Current:** `text-sm font-semibold text-muted-foreground uppercase tracking-wide mb-3`

**Issue:** `text-sm` (14px) uppercased is too large for a section label. Premium apps use 10-11px for uppercase labels. At 14px, uppercase text looks like a shouting header, not a quiet section divider.

**Recommended:** `text-[11px] font-semibold text-muted-foreground uppercase tracking-wider mb-3`

This change applies to: WeeklySummary section headers, and any future section headers in other screens.

### Nav bar

**Current:**
```jsx
<nav className="fixed bottom-0 left-0 right-0 border-t bg-background">
  <div className="flex">
    <button className="flex-1 py-3 text-sm font-medium text-center ...">
```

**Issues:**
1. No visual weight separating the nav from content — `border-t` exists but `bg-background` is identical to the content background, making the nav "float."
2. Tab text size (`text-sm`) matches body text — active/inactive distinction relies only on color.

**What premium apps do:** Bottom nav bars in macOS-adjacent apps (Things 3, Craft) have a very slight background tint or increased opacity that grounds them. They also commonly use `text-xs` for nav labels with `text-sm` for active state — or use the same size but increase weight for active.

**Recommended:**
```jsx
<nav className="fixed bottom-0 left-0 right-0 border-t border-border bg-background/95 backdrop-blur-none">
  <div className="flex">
    <button className={`flex-1 py-3 text-[13px] font-medium text-center ...`}>
```

Note: `backdrop-blur` is tempting but has a performance cost in Electron and may cause rendering artifacts. Use `bg-background/95` instead (very slightly transparent background) — this creates the "grounded" feel without blur cost.

### Overdue badge

**Current:** `text-xs text-muted-foreground bg-muted rounded px-2 py-0.5`

**Issue:** `bg-muted` is the same color as the hover state. The badge blends into the hover background when the row is hovered.

**Recommended:** `text-xs text-muted-foreground bg-muted/70 rounded-sm px-2 py-0.5`

The `rounded-sm` (instead of `rounded`) is subtly more refined — matches Linear's badge corner radius style.

### EmptyState (critical fix)

**Current:**
```jsx
<div className="flex flex-col items-center justify-center h-64">
  <h2 className="text-lg font-semibold text-foreground">All clear — nothing left for today.</h2>
  <p className="text-sm text-muted-foreground mt-2">Add a task above to get started.</p>
</div>
```

**Problem:** The entire TodayView is flush-left at `px-4` (TaskRow) and `px-6` (header). EmptyState centers itself in the middle of the screen, breaking the layout language. This is the most visually jarring inconsistency in the app.

**What premium apps do:** Notion, Linear — empty states are left-aligned within the content column, not centered. Centering is reserved for full-page empty states (no sidebar, no nav). For inline empty states (within a list area), left-align matching the content baseline.

**Recommended:**
```jsx
<div className="px-4 py-8">
  <p className="text-sm text-muted-foreground">All clear — nothing left for today.</p>
  <p className="text-[13px] text-muted-foreground mt-1">Add a task to get started.</p>
</div>
```

This flattens the hierarchy (no `text-lg` heading), left-aligns with `px-4` (matching TaskRow), and adds `py-8` breathing room. The empty state should be quiet, not prominent.

**Confidence:** HIGH.

## Research Area 5: Motion & Micro-interactions

### Constraint

PROJECT.md explicitly states: **"No animations required; prioritize clarity over aesthetics."** Phase 04.1 CONTEXT.md confirms: "No animations — PROJECT.md constraint; do NOT add transition classes."

This means: **no `transition`, `duration`, `ease`, `animate-*`, or CSS `animation` properties anywhere.**

### What to do instead

Premium feel without animation comes from:
1. **Crisp state changes** — hover states that are visually distinct enough that the instant switch feels intentional, not broken
2. **Correct cursor usage** — `cursor-pointer` on all interactive elements (already in use), `cursor-not-allowed` on disabled elements (time input already has this)
3. **Immediate feedback** — inline error states, immediate visual response to clicks

**The checkbox hover pattern** described in the Component section above (`hover:border-primary hover:bg-primary/10`) achieves affordance signaling without animation.

**Confidence:** HIGH — constraint is documented. Research confirms premium feel is achievable without animation through better static states.

## Research Area 7: Tailwind + Electron Implementation Notes

### Font loading — offline safety

Using `@fontsource/inter` loads fonts from `node_modules` as local CSS imports. This is:
- Offline-safe (no CDN)
- Compatible with Electron's file:// protocol (fonts served from asar)
- Correct approach for this project

**Alternative (Google Fonts CDN):** Do NOT use. PROJECT.md requires zero network access.

### Tailwind font-family extension

```js
// tailwind.config.js
theme: {
  extend: {
    fontFamily: {
      sans: ['Inter', 'system-ui', '-apple-system', 'BlinkMacSystemFont', 'sans-serif'],
    },
    fontSize: {
      // Add custom 13px size for meta/label text
      '2xs': ['11px', { lineHeight: '1rem' }],
      'ui': ['13px', { lineHeight: '1.25rem' }],
    },
  },
}
```

This adds `text-2xs` (11px for section labels) and `text-ui` (13px for meta text) as named tokens instead of scattered `text-[13px]` arbitrary values.

### CSS custom properties for font-feature-settings

```css
@layer base {
  :root {
    font-feature-settings: 'cv11' 1, 'tnum' 1, 'liga' 1, 'calt' 1;
    -webkit-font-smoothing: antialiased;
    -moz-osx-font-smoothing: grayscale;
  }
}
```

`-webkit-font-smoothing: antialiased` is important for Electron on macOS — without it, text renders with slightly fuzzy subpixel rendering even when macOS has disabled subpixel antialiasing system-wide. This is the standard fix used in VSCode, Linear webapp, and other Electron/web apps.

**Confidence:** HIGH — `-webkit-font-smoothing: antialiased` is a verified standard in the Electron community (confirmed via multiple Electron issues and all major Electron app source code).

### Electron window chrome considerations

**Current app:** Uses the default Electron titlebar (native OS chrome). This is the correct choice for a productivity app targeting macOS and Windows.

**What NOT to do:** Do not add `titleBarStyle: 'hiddenInset'` or custom frameless window — this is a significant scope change, creates platform-inconsistency, and requires `-webkit-app-region: drag` CSS throughout the header. The risk/reward is poor.

**What IS worth doing:** The header area (`px-6 py-6`) of TodayView sits directly below the native OS titlebar. A `border-b border-border` between the TodayView header and the task list creates visual separation that makes the transition from OS chrome to app content feel cleaner.

### Tailwind purging / content scanning

Current `tailwind.config.js` content path: `'./src/renderer/**/*.{ts,tsx,html}'` — this correctly captures all renderer files. Any new CSS variables added to `index.css` are automatically available without config changes.

### shadcn component compatibility

All shadcn components use the CSS variable tokens already. Adding Inter font affects them automatically via the `font-sans` Tailwind class which shadcn applies to `body`. No changes needed to `button.tsx`, `input.tsx`, `dialog.tsx`, etc.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Offline Inter font serving | Custom font file copying/loading | `@fontsource/inter` npm package | Already handles format negotiation, CSS imports, tree shaking |
| Custom checkbox component | SVG checkbox with JS state | CSS-only hover state on existing button | No animation constraint; CSS is sufficient |
| Fluid type scale | CSS `clamp()` responsive type | Named Tailwind tokens (static) | Desktop app, fixed viewport range — fluid type adds complexity with no benefit |
| Dark mode detection | JS media query listener | `darkMode: ['class']` already in Tailwind config | Already wired; `.dark` class on `html` element triggers all dark tokens |

**Key insight:** The gap between this app's current look and a premium look is almost entirely font and spacing — not component complexity. Resist the urge to add new components.

## Common Pitfalls

### Pitfall 1: Adding `transition` classes under the heading of "polish"
**What goes wrong:** Developer sees a hover state and adds `transition-colors duration-150` thinking it feels more premium. This directly violates the PROJECT.md constraint and the Phase 04.1 CONTEXT.md decision.
**Why it happens:** Motion polish is instinctive when improving UI.
**How to avoid:** CTRL+F for `transition` and `duration-` in any PR diff. If found, remove.
**Warning signs:** "This just needs a subtle transition" thinking.

### Pitfall 2: font-feature-settings not cascading correctly
**What goes wrong:** Setting `font-feature-settings` in `:root` and then adding another `font-feature-settings` declaration on a child element replaces (not merges) the root settings.
**Why it happens:** CSS `font-feature-settings` does not cascade/merge — it replaces entirely.
**How to avoid:** Set all feature flags in one declaration on `:root`. If a specific component needs different settings, include ALL desired flags in that component's declaration, not just the additions.
**Warning signs:** Tabular numbers stop working inside a component that has its own font-feature-settings.

### Pitfall 3: `@fontsource/inter` and asar packaging
**What goes wrong:** Font CSS files import font binaries using relative paths. When Electron packages the app, the font binaries may be inside the asar archive but the CSS paths may break.
**Why it happens:** Vite resolves `@fontsource/inter` imports at build time and bundles the fonts into the output — this usually works correctly, but only if the import is in a file that Vite processes (CSS or JS, not a raw HTML file).
**How to avoid:** Import `@fontsource/inter/400.css` from `src/renderer/index.css` (already the entry point for styles). Vite handles it. Do NOT add a `<link>` tag in `index.html` pointing to node_modules.
**Warning signs:** Fonts work in development but fall back to system font after `npm run package`.

### Pitfall 4: Breaking the two-touchpoint indigo rule
**What goes wrong:** While improving components, developer accidentally adds `text-primary` or `border-primary` to a third element (e.g., styling the Stats page numbers in indigo for "impact").
**Why it happens:** More color feels more polished to non-designers.
**How to avoid:** Search for `text-primary` and `border-primary` in the diff and count occurrences. Should be exactly 2: the Add Task button and the active nav indicator.
**Warning signs:** Any new `primary` usage not in Add Task button or nav active state.

### Pitfall 5: Text size regression in dark mode
**What goes wrong:** Changing font from system default to Inter at 14px may make dark mode text appear slightly thinner due to how Chromium renders Inter at that weight in dark mode.
**Why it happens:** Inter at weight 400 can appear visually thinner than system-ui (SF Pro) on macOS dark mode.
**How to avoid:** Test dark mode explicitly after adding Inter. If thin, bump body text from `font-normal` (400) to a weight between 400 and 500 — in CSS: `font-weight: 420` (variable font value, Inter supports this).
**Warning signs:** User reports "text looks light/thin" after font change in dark mode.

## Architecture Patterns

### Recommended change scope for this phase

This phase touches only CSS, CSS variables, and Tailwind class strings. It does NOT:
- Add new components

## Open Questions

1. **Checkbox hover using indigo**
   - What we know: Phase 04.1 locked indigo to exactly 2 touchpoints (Add Task button, active nav). Checkbox hover-only usage (`hover:border-primary hover:bg-primary/10`) is transient and not persistent.
   - What's unclear: Does a transient hover state count as a "touchpoint" under the Phase 04.1 rule?
   - Recommendation: Implement as `hover:border-muted-foreground hover:bg-muted` (neutral hover) to stay clearly within the rule. Revisit if user testing shows the affordance is weak.

2. **Warmer foreground color**
   - What we know: Linear moved to warmer grays. Current `--foreground: 222.2 84% 4.9%` is cold.
   - What's unclear: Whether users notice or care about the warmth of near-black text.
   - Recommendation: Leave `--foreground` unchanged for this phase (lowest-risk approach). If a future design review wants warmth, do it as a standalone token-only change.

3. **`@fontsource/inter` and asar packaging**
   - What we know: Vite processes `@fontsource/inter` CSS imports correctly in development. Electron packaging with asar may require testing.
   - What's unclear: Whether Electron Forge + Vite correctly bundles font binary files from `@fontsource/inter` into the asar.
   - Recommendation: Treat this as the one area requiring explicit verification during implementation. If font fails in packaged mode, fallback is `@import url()` from a local `fonts/` directory (copy font files manually).
